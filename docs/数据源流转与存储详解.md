# 数据源流转与存储详解

## 概述

本文档详细说明 GMGN 插件中三个数据源的流转过程、存储结构和优先级关系。

## 三个数据源

### 1. GMGN Holder API（持有人数据）
- **触发方式**：插件设置页面点击"持有人启动"
- **数据特点**：完整的持有人列表，包含持仓量、资金来源等
- **优先级**：最高（最完整的数据）

### 2. GMGN Trade API（交易数据）
- **触发方式**：插件设置页面点击"活动监听启动"
- **数据特点**：交易记录，包含买卖行为、金额、时间等
- **优先级**：中等（补充交易信息）

### 3. Helius WebSocket API（实时数据）
- **触发方式**：自动连接，实时监听链上交易
- **数据特点**：实时交易事件，延迟最低
- **优先级**：最低（实时补充，但信息不如前两者完整）

## 数据流转图

### 流程 1: 持有人启动（GMGN Holder API）

```
用户点击"持有人启动"
    ↓
[index.jsx] hookHolderHandler 拦截 GMGN API 响应
    ↓
提取 holders 数组（包含 owner, balance, funding_account 等）
    ↓
window.__heliusIntegration.updateGmgnHolders(holders)
    ↓
[HeliusIntegration.js] updateGmgnHolders()
    ↓
monitor.updateHolderData(holders)
    ↓
[HeliusMonitor.js] updateHolderData()
    ├─ 步骤 1: metricsEngine.updateUsersInfo(holders)
    │   ↓
    │   [MetricsEngine.js] updateUsersInfo()
    │   ├─ 遍历 holders 数组
    │   ├─ 为每个 holder 创建/更新 userInfo[address]
    │   └─ 设置 data_source = 'GMGN Holder API'
    │
    ├─ 步骤 2: scoringEngine.calculateScores()
    │   ↓
    │   [ScoringEngine.js] 计算每个用户的分数
    │   └─ 返回 scoreMap（address → {score, reasons, isWhale}）
    │
    ├─ 步骤 3: 将分数存储到 userInfo
    │   └─ userInfo[address].score = scoreData.score
    │
    ├─ 步骤 4: 根据 scoreThreshold 过滤用户
    │   └─ filteredUsers = users where score < threshold
    │
    └─ 步骤 5: 重新计算指标
        └─ metricsEngine.getMetrics()
```

### 流程 2: 活动监听启动（GMGN Trade API）

```
用户点击"活动监听启动"
    ↓
[index.jsx] hookTradesHandler 拦截 GMGN API 响应
    ↓
提取 trades 数组（包含 owner, type, amount, timestamp 等）
    ↓
window.__heliusIntegration.updateGmgnTrades(trades)
    ↓
[HeliusIntegration.js] updateGmgnTrades()
    ↓
monitor.updateTradeData(trades)
    ↓
[HeliusMonitor.js] updateTradeData()
    ├─ metricsEngine.updateTradeStats(trades)
    │   ↓
    │   [MetricsEngine.js] updateTradeStats()
    │   ├─ 遍历 trades 数组
    │   ├─ 更新 traderStats[address]（买入/卖出统计）
    │   └─ 如果 userInfo[address] 不存在，创建并设置 data_source = 'GMGN Trade API'
    │
    └─ 重新计算指标
        └─ metricsEngine.getMetrics()
```

### 流程 3: Helius 实时数据（WebSocket）

```
Helius WebSocket 接收到交易事件
    ↓
[HeliusMonitor.js] processTransaction()
    ↓
解析交易数据（from, to, amount, type）
    ↓
metricsEngine.updateUserFromHelius(address, txData)
    ↓
[MetricsEngine.js] updateUserFromHelius()
    ├─ 如果 userInfo[address] 不存在，创建并设置 data_source = 'Helius API'
    ├─ 更新交易统计
    └─ 更新 traderStats[address]
```

## 数据源优先级和合并逻辑

### 优先级规则

1. **GMGN Holder API** > **GMGN Trade API** > **Helius API**
2. 如果用户已存在，不会覆盖 data_source（保留最高优先级的来源）
3. 但会合并和更新其他字段（如交易统计）

### 合并逻辑示例

```javascript
// 场景 1: 先收到 Helius 数据，后收到 Holder 数据
// 1. Helius 创建用户
userInfo["addr1"] = {
  owner: "addr1",
  data_source: "Helius API",
  buy_count: 1
};

// 2. Holder API 更新用户
userInfo["addr1"] = {
  owner: "addr1",
  data_source: "GMGN Holder API",  // 更新为更高优先级
  balance: 1000000,                 // 新增持仓信息
  funding_account: "fund1",         // 新增资金来源
  buy_count: 1                      // 保留交易统计
};

// 场景 2: 先收到 Holder 数据，后收到 Trade 数据
// 1. Holder API 创建用户
userInfo["addr2"] = {
  owner: "addr2",
  data_source: "GMGN Holder API",
  balance: 500000
};

// 2. Trade API 更新用户
userInfo["addr2"] = {
  owner: "addr2",
  data_source: "GMGN Holder API",  // 保持最高优先级
  balance: 500000,                  // 保留持仓信息
  buy_count: 3,                     // 新增交易统计
  sell_count: 1
};
```

    const address = holder.owner;

}

## 数据源标识的作用

1. **调试追踪**：快速定位数据来自哪个 API
2. **数据质量评估**：Holder API 数据最完整，Helius 数据最实时
3. **问题排查**：如果某个用户数据不完整，可以查看数据源判断原因
4. **优先级管理**：确保高质量数据不被低质量数据覆盖

## 常见问题

### Q1: 为什么有些用户只有交易数据，没有持仓数据？

**A**: 该用户的数据来源是 "GMGN Trade API" 或 "Helius API"，这两个数据源不包含持仓信息。只有 "GMGN Holder API" 才提供完整的持仓数据。

### Q2: 如果同一个用户从多个数据源获取数据，会怎样？

**A**: 数据会合并，但 data_source 字段会保留最高优先级的来源（Holder > Trade > Helius）。例如，先从 Helius 获取交易数据，后从 Holder API 获取持仓数据，最终 data_source 会显示 "GMGN Holder API"。

### Q3: 如何确保获取最完整的数据？

**A**: 建议按以下顺序启动：
1. 先点击"持有人启动"（获取完整持仓数据）
2. 再点击"活动监听启动"（补充交易数据）
3. Helius 会自动连接（实时补充）

### Q4: data_source 字段会实时更新吗？

**A**: 会。当更高优先级的数据源提供数据时，data_source 会更新。例如，用户最初从 Helius 获取数据（data_source = "Helius API"），后来 Holder API 提供了该用户的数据，data_source 会更新为 "GMGN Holder API"。

## 总结

- **三个数据源**：GMGN Holder API（最完整）、GMGN Trade API（交易数据）、Helius API（实时数据）
- **统一存储**：所有数据存储在 MetricsEngine.userInfo 中
- **优先级管理**：Holder > Trade > Helius，确保高质量数据不被覆盖
- **数据源标识**：每个用户都有 data_source 字段，标识数据来源

## 混合数据源策略（v2.0 新增）

### 背景

GMGN Holder API 只能返回前 100 个持有者，导致其他用户信息不全。为了解决这个问题，实施了混合数据源策略，结合 Holder API、Trade API 和 Helius API 来获取更完整的用户数据。

### 两种数据模式

#### 1. holder_based 模式（基于 Holder 快照）

**特点**：
- 有完整的 Holder API 数据作为基准
- 持仓信息准确（来自 Holder API 快照）
- Trade 数据增量更新到快照上

**适用场景**：
- 用户在 Holder API 返回的前 100 名内
- 先启动"持有人启动"，后启动"活动监听"

**数据流程**：
```
1. 获取 Holder API 数据
   ↓
2. 初始化 userInfo[address]
   - data_mode = 'holder_based'
   - has_holder_snapshot = true
   - ui_amount = holder.ui_amount（准确值）
   ↓
3. 初始化 traderStats[address]
   - netTokenReceived = holder.ui_amount
   ↓
4. 收到新 Trade 数据
   ↓
5. 增量更新（updateTradeIncremental）
   - netTokenReceived += trade.token_amount（buy）
   - netTokenReceived -= trade.token_amount（sell）
   - 更新 totalBuySol, totalSellSol
   ↓
6. 指标计算
   - 根据 netTokenReceived 判断是否还持仓
   - 计算已落袋、本轮下注、浮盈浮亏等
```

#### 2. trade_calculated 模式（从 Trade 计算）

**特点**：
- 只有 Trade 数据，没有 Holder 快照
- 持仓信息从交易记录计算（可能不准确）
- 每次新交易都从头计算

**适用场景**：
- 用户不在 Holder API 返回的前 100 名内
- 先启动"活动监听"，后启动"持有人启动"（或不启动）

**数据流程**：
```
1. 收到 Trade 数据
   ↓
2. 添加到 TradeCache
   ↓
3. 从头计算（calculateFromTrades）
   - 遍历用户的所有交易
   - netTokenReceived = Σ(buy) - Σ(sell)
   - 计算 totalBuySol, totalSellSol
   ↓
4. 更新 userInfo[address]
   - data_mode = 'trade_calculated'
   - has_holder_snapshot = false
   - ui_amount = netTokenReceived（计算值）
   ↓
5. 指标计算
   - 根据 netTokenReceived 判断是否还持仓
   - 计算已落袋、本轮下注、浮盈浮亏等
```

### 模式切换

当用户从 trade_calculated 模式切换到 holder_based 模式时：

```
1. 用户最初只有 Trade 数据
   - data_mode = 'trade_calculated'
   - has_holder_snapshot = false
   - ui_amount = 500000（从交易计算）
   ↓
2. 用户持仓增加，进入前 100 名
   ↓
3. Holder API 返回该用户数据
   ↓
4. 更新用户信息（updateUserInfo）
   - data_mode = 'holder_based'（切换模式）
   - has_holder_snapshot = true
   - ui_amount = 1200000（Holder API 的准确值）
   - last_holder_update = Date.now()
   ↓
5. 更新 traderStats
   - netTokenReceived = 1200000（使用准确值）
   ↓
6. 之后的新 Trade 使用增量更新
   - 不再从头计算
   - 直接在 1200000 基础上增减
```

### TradeCache 管理

**职责**：
- 管理所有 Trade 数据
- 按用户分组存储
- 去重和时间排序

**关键方法**：
```javascript
// 添加交易并去重
const isNew = tradeCache.addTrade(trade);

// 获取用户的所有交易
const trades = tradeCache.getAllTradesForUser(address);

// 检查用户是否有交易记录
const hasTrades = tradeCache.hasTradesForUser(address);
```

### 指标计算支持

混合数据源策略完全支持 Helius 实时指标的计算：

**关键指标**：
1. **已落袋**：已退出用户（netTokenReceived < 1）的实现盈亏
2. **本轮下注**：活跃用户（netTokenReceived >= 1）的买入总额
3. **浮盈浮亏**：活跃用户的未实现盈亏
4. **活跃用户数**：netTokenReceived >= 1 的用户数
5. **已退出用户数**：netTokenReceived < 1 的用户数

**两种模式都支持**：
- holder_based 模式：netTokenReceived 基于 Holder 快照 + 增量更新，准确度高
- trade_calculated 模式：netTokenReceived 从交易计算，可能有误差但有数据

### 数据结构更新

**新增字段**：
```javascript
userInfo[address] = {
  // ... 现有字段 ...
  data_mode: 'holder_based' | 'trade_calculated',  // 数据模式
  has_holder_snapshot: boolean,                     // 是否有 Holder 快照
  last_holder_update: timestamp,                    // 最后 Holder 更新时间
  last_trade_update: timestamp,                     // 最后 Trade 更新时间
  calculated_balance: number,                       // 从 Trade 计算的持仓
}
```

### 使用建议

**场景1：追求数据准确性**
1. 先点击"持有人启动"（获取前 100 名的准确数据）
2. 再点击"活动监听启动"（增量更新）
3. 结果：前 100 名用户数据准确，其他用户从交易计算

**场景2：追求数据完整性**
1. 先点击"活动监听启动"（获取所有交易用户）
2. 再点击"持有人启动"（前 100 名切换到准确模式）
3. 结果：所有有交易的用户都有数据，前 100 名更准确

**场景3：只关注大户**
1. 只点击"持有人启动"
2. 结果：只显示前 100 名持有者，数据最准确
- **用户可见**：在用户详情面板中显示数据来源，方便调试和理解
