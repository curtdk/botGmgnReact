# Helius 高性能实时架构与 Rate Limit 优化报告

本文档记录了基于 Helius 全栈（RPC + WSS）的高性能数据同步架构设计，以及针对 Helius Free Tier 严格 Rate Limit 的优化措施。

## 1. 架构概览 (Architecture)

为了实现"首次全量极速同步 + 实时毫秒级更新"，我们采用了以下混合架构：

### A. 历史数据层 (History Layer)
*   **Signatures Fetch**: 使用 `getSignaturesForAddress` (Limit 1000) 分页拉取全量历史签名。
*   **Batch Parsing**: 使用并发的 `getTransaction` (Encoded: jsonParsed) 补全交易详情。
    *   *优化*: 由于 Free Tier 不支持 Batch RPC 请求，改为**并发单请求 (Concurrency Control)** 模式。

### B. 实时数据层 (Real-time Layer)
*   **WebSocket**: 连接 Helius WSS (`logsSubscribe`)，监听目标 Mint 的所有活动。
*   **Buffer Queue**: 收到的实时 `signature` 暂存入队列，等待历史同步完成后处理。
*   **Deduplication**: 利用内存 Set 对签名去重，防止历史拉取与实时推送的数据冲突。

### C. 计算引擎 (Metrics Engine)
*   **Incremental Calculation**: 状态机模式，接收标准化 `ParsedTransaction` 对象，增量更新用户持仓和全局指标。

## 2. 关键优化：应对 429 Rate Limit

在测试过程中，我们发现 Helius Free Tier 对 API 调用频率有严格限制（尤其是并发请求）。为此实施了多层防护：

### 1. 指数退避重试 (Exponential Backoff)
在 `rpc_client.js` 中实现了智能重试机制：
*   检测到 `429 Too Many Requests` 或 `-32429` 错误时，自动暂停并重试。
*   **退避策略**: 初始等待 1s -> 2s -> 4s，最多重试 3 次。

### 2. 并发控制 (Concurrency Throttling)
在 `data_fetcher.js` 中严格限制并发数：
*   **Batch Size**: 降至 **5** (即同时只发 5 个请求)。
*   **Interval Delay**: 每批请求后强制等待 **200ms**，给服务器缓冲时间。
*   **动态降级**: 如果检测到 429，自动增加等待时间至 5s。

## 3. 运行指南

### 启动监控 (Monitoring)

支持两种启动方式：

**方式一：命令行参数（推荐）**
直接在命令中指定要监控的 Mint 地址。
```bash
node scripts/helius_full_monitor.js <MINT_ADDRESS>
```
示例：
```bash
node scripts/helius_full_monitor.js 9svdK1bjBBuk1tqmeqSHrVSaD6M5wqLEsvFmG9SFpump
```

**方式二：交互式输入**
直接运行脚本，按提示输入地址。
```bash
node scripts/helius_full_monitor.js
# 提示: 请输入要监控的 Mint 地址 (回车使用默认): ...
```

### 多实例运行
如果需要同时监控多个代币，只需打开多个终端窗口，分别运行上述命令即可。系统会为每个 Mint 创建独立的缓存目录（`data/cache/{mint}/`），互不干扰。

### 预期输出
1.  **[History]**: 显示历史签名拉取进度。
2.  **[WSS]**: 显示 WebSocket 连接状态。
3.  **[Metrics]**: 历史同步完成后，打印初始指标。
4.  **[Realtime]**: 随后每秒检查新交易，有更新时自动打印新指标。

## 4. 性能数据 (基于 Free Tier)
*   **历史同步速度**: 受限于 429 防护，目前约为 **15-20 TPS** (Transactions Per Second)。
    *   *注*: 付费版 Helius 支持 Batch Request，速度可提升 10-50 倍。
*   **实时延迟**: WSS 推送延迟 < 500ms，处理延迟 < 1s。

## 5. 代码结构
*   `scripts/helius_core/rpc_client.js`: 包含重试逻辑的 RPC 客户端。
*   `scripts/helius_core/data_fetcher.js`: 包含并发控制的数据获取器。
*   `scripts/helius_core/metrics_engine.js`: 核心指标计算逻辑。
*   `scripts/helius_full_monitor.js`: 主控程序。

## 6. 常见问题
*   **WSS 429 Error**: 如果启动时 WSS 连接报错 429，说明 IP 请求过多。程序会自动重试，通常等待 10-30s 后恢复。
*   **Batch requests are only available for paid plans**: 已通过改为并发单请求解决。
