# Helius API 全量数据算法与性能分析报告

本报告记录了使用 Helius API 获取代币全量历史数据并计算核心盈亏指标的方法、性能统计及结果。

## 1. 核心算法：全量回放 (Genesis Replay)

为了准确计算“已落袋”、“本轮成本”等与历史投入相关的指标，我们采用了**全量回放**算法。

### 为什么必须全量？
*   **成本计算依赖历史**: 用户的持仓成本（Cost Basis）是由其历史买入价格决定的。如果只获取最近的数据（如最近100笔），我们就不知道用户最初花了多少钱买入，从而无法计算准确的盈亏。
*   **已落袋依赖离场数据**: “已落袋”指标是指已经卖出清仓的用户的利润。这部分用户可能在很久以前就买入并卖出了，只有拉取全量历史才能捕捉到他们的完整行为。

### 算法步骤
1.  **全量拉取 (Fetch All)**: 使用 API 的分页功能，从最新一笔交易一直追溯到代币创建的第一笔交易 (Genesis)。
2.  **正序排列 (Sort ASC)**: 将获取到的交易按时间**正序**（旧 -> 新）排列。
3.  **事件回放 (Replay)**: 在内存中建立一个“账本”，模拟每一笔交易对用户资产的影响：
    *   **买入**: 增加用户的 Token 持仓，增加 SOL 投入成本。
    *   **卖出**: 减少用户的 Token 持仓，增加 SOL 回收额（实现利润）。
4.  **指标快照**: 遍历结束后的最终状态即为当前时刻的准确指标。

## 2. API 使用说明

### 接口详情
*   **Endpoint**: `https://api.helius.xyz/v0/addresses/{mint}/transactions`
*   **Method**: `GET`
*   **Params**:
    *   `api-key`: 您的 Helius API Key。
    *   `before`: 分页游标，传入上一页最后一条交易的 `signature` 以获取更早的数据。

### 过滤逻辑
为了确保数据纯净，我们在拉取时自动过滤了失败交易：
```javascript
// 仅保留 transactionError 和 err 字段为空的交易
const validTxs = data.filter(tx => !tx.transactionError && !tx.err);
```

### 返回值结构与中文注释
API 返回的 `EnhancedTransaction` 对象关键字段如下：

```json
{
  "signature": "...", // 交易哈希 (Tx Hash)
  "timestamp": 1700000000, // 交易时间戳
  "type": "SWAP", // 交易类型
  "transactionError": null, // **关键**: 交易错误信息，为 null 表示成功
  "feePayer": "UserAddress...", // 交易发起人 (通常是 Trader)
  
  // 账户资金变动数据 (核心计算依据)
  "accountData": [
    {
      "account": "UserAddress...",
      "nativeBalanceChange": -1000000000, // **SOL 变动**: 负数代表支出(买入), 正数代表收入(卖出) (单位: Lamports)
      "tokenBalanceChanges": [
        {
          "mint": "MintAddress...",
          "rawTokenAmount": {
            "tokenAmount": "1000000", // **Token 变动**: 正数代表获得, 负数代表卖出
            "decimals": 6
          }
        }
      ]
    }
  ]
}
```

## 3. 性能与结果分析

本次测试针对 Mint: `9svdK1bjBBuk1tqmeqSHrVSaD6M5wqLEsvFmG9SFpump`

### 性能统计
*   **总有效交易数**: 1,491 笔
*   **API 拉取耗时**: 18.95 秒 (约 15 个分页请求)
*   **计算回放耗时**: < 0.01 秒 (内存计算极快)
*   **总耗时**: 约 19 秒

### 计算结果 (四大指标)
基于全量历史数据的精准计算结果：

1.  **已落袋 (Realized PnL)**: `6.2988 SOL`
    *   *解释*: 所有已清仓离场用户赚取的净利润总和。
2.  **本轮下注 (Current Bet)**: `37.7325 SOL`
    *   *解释*: 当前所有持有者买入这些代币所花费的 SOL 总额（总投入）。
3.  **本轮成本 (Current Cost)**: `18.7189 SOL`
    *   *解释*: 当前持有者的净投入（总买入 - 总卖出）。这意味着持有者通过部分卖出已经收回了约 19 SOL 的成本。
4.  **浮盈浮亏 (Floating PnL)**: `-13.6754 SOL`
    *   *解释*: 当前持有者的未实现盈亏。
    *   *计算*: (当前持仓市值) - (净投入成本)。
    *   *状态*: 负数表示当前持有者总体处于浮亏状态。

## 4. 结论
使用“全量拉取 + 正序回放”方案是可行的，对于中小型代币（几千笔交易）耗时在可接受范围内（< 20秒）。此方法能提供最准确的盈亏分析。
