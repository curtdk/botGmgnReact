# Helius 算法与缓存机制详解

本文档详细解释了指标计算的核心算法以及新引入的本地缓存机制。

## 1. 本地缓存机制 (Caching System)

为了解决 Helius API 限流问题并提高重启速度，我们实现了一套基于文件的持久化缓存。

### 存储结构
所有数据存储在 `data/cache/{mint_address}/` 目录下：
1.  **`signatures.json`**: 存储已获取的交易签名列表（数组）。
2.  **`transactions.jsonl`**: 存储交易的详细 JSON 数据（JSON Lines 格式，每行一个 JSON 对象）。

### 增量更新逻辑 (Incremental Fetch)
每次程序启动或同步历史时，执行以下步骤：

1.  **加载缓存**: 读取 `signatures.json`。
    *   *优化*: 如果存在 1000 条缓存签名，程序知道这 1000 条不需要再下载。
2.  **获取最新签名 (Until)**:
    *   调用 `getSignaturesForAddress(mint, { until: 最新缓存签名 })`。
    *   API 仅返回比"最新缓存签名"更新的交易。
    *   例如：链上有 1500 条交易，缓存了 1490 条。API 只返回最新的 10 条。
3.  **下载新数据**: 仅对这 10 条新签名调用 `getTransaction` 下载详情。
4.  **追加保存**:
    *   新签名追加到 `signatures.json`。
    *   新详情追加写入 `transactions.jsonl`。

**优势**:
*   **极速重启**: 第二次运行几乎是瞬时的（只需加载本地文件）。
*   **节省额度**: 不再重复请求旧数据，Credits 消耗降低 99%。

## 2. 指标计算算法 (Metrics Algorithm)

我们使用**全量回放 (Genesis Replay)** 算法来计算四大指标。这意味着我们从代币诞生的第一秒开始，模拟每一笔交易对每个用户钱包的影响。

### 核心状态
系统维护一个全局账本 `traderStats`，记录每个用户的：
*   `netSolSpent`: 净 SOL 投入（买入 SOL - 卖出 SOL）。如果是负数，说明用户已经回本并盈利。
*   `netTokenReceived`: 净 Token 持仓。
*   `totalBuySol`: 累计买入 SOL 总额。
*   `totalSellSol`: 累计卖出 SOL 总额。

### 交易处理逻辑 (Step-by-Step)

对于每一笔交易（按时间正序）：
1.  **识别买卖**:
    *   如果 SOL 减少且 Token 增加 -> **买入**。
    *   如果 SOL 增加且 Token 减少 -> **卖出**。
2.  **更新状态**:
    *   **买入**: `netSolSpent` 增加，`totalBuySol` 增加。
    *   **卖出**: `netSolSpent` 减少，`totalSellSol` 增加。

### 四大指标计算公式

当所有历史交易回放完毕后，根据最终状态计算指标：

#### 1. 已落袋 (Realized PnL of Exited Users)
*   **定义**: 已经清仓离场（持仓 ≈ 0）的用户的净利润。
*   **公式**: `∑ (离场用户的 TotalSell - 离场用户的 TotalBuy)`
*   **代码对应**: `metrics_engine.js` 中的 `yiLuDai`。

#### 2. 本轮下注 (Current Bet)
*   **定义**: 当前持有者（持仓 > 0）累计投入的本金总额。
*   **公式**: `∑ (当前持有者的 TotalBuy)`
*   **代码对应**: `metrics_engine.js` 中的 `benLunXiaZhu`。

#### 3. 本轮成本 (Current Cost / Net Investment)
*   **定义**: 当前持有者的净投入成本（扣除部分卖出后的剩余成本）。
*   **公式**: `本轮下注 - 当前持有者的已实现卖出额`
*   **逻辑**: 如果我买了 10 SOL，卖了 2 SOL，我的本轮下注是 10，但我的"在场成本"（Net Cost）变成了 8。
*   **代码对应**: `metrics_engine.js` 中的 `benLunChengBen`。

#### 4. 浮盈浮亏 (Floating PnL)
*   **定义**: 当前持仓的未实现盈亏。
*   **公式**: `(当前持仓量 * 当前价格) - 净投入成本`
*   **代码对应**: `metrics_engine.js` 中的 `floatingPnL`。

### 举例说明
用户 Bob 的操作：
1.  买入 1000 Token，花费 10 SOL。
    *   `netSolSpent`: 10, `TotalBuy`: 10.
2.  卖出 500 Token，获得 12 SOL (价格暴涨)。
    *   `netSolSpent`: 10 - 12 = -2 (负成本，意味着已无风险且盈利 2 SOL)。
    *   `TotalSell`: 12.
    *   持仓: 500.

**此时 Bob 对指标的贡献**:
*   **已落袋**: 0 (Bob 还没清仓，所以不算已落袋)。
*   **本轮下注**: 10 SOL.  
*   **本轮成本**: -2 SOL (因为他已经提走了比本金更多的钱)。
*   **浮盈浮亏**: (500 * 价格) - (-2) = 市值 + 2。

这个算法能最真实地反映资金流向和持仓心理。
