# 庄家档案库（Boss Profile Archive）

## 概述

庄家档案库是一个跨 mint 的持久化存储系统。当用户在任意 mint 页面被评分后，得分 > 0 的用户记录会被保存到本地。下次在新的 mint 页面遇到同一地址时，系统可以直接利用历史数据，无需重新执行检测。

---

## 存储结构

**Storage Key**：`boss_profiles`（`chrome.storage.local`，全局单一 key）

```json
{
  "钱包地址": {
    "status": "庄家 | 散户",
    "score": 25,
    "score_reasons": ["无来源(+10)", "隐藏中转[Create+CloseAccount](+15)"],
    "first_seen": 1772025060450,
    "last_seen": 1772025178432,
    "seen_count": 3,
    "last_mint": "QSsF56toZUnyZf6EXubnm49ZxMoyXbpa6yugBkhpump"
  }
}
```

| 字段 | 说明 |
|---|---|
| `status` | 保存时的状态（由 statusThreshold 决定，可能和得分不完全对应） |
| `score` | 保存时的总分 |
| `score_reasons` | 各规则得分原因列表 |
| `first_seen` | 首次被检测到的时间戳 |
| `last_seen` | 最近一次检测时间戳 |
| `seen_count` | 出现次数（每次评分 +1） |
| `last_mint` | 最近出现在哪个 mint |

## 保存逻辑（saveBossProfiles）

**文件**：`src/helius/HeliusMonitor.js`

**触发时机**：`updateHolderData()` 完成评分后（步骤7之后），fire-and-forget 调用，不阻塞主流程。

**保存条件**：`score > 0`（score = 0 的用户不保存，节省空间）

**合并策略**：
- 同一地址再次出现时，更新 `last_seen`、`seen_count`、`score`、`score_reasons`、`status`
- `first_seen` 保留首次记录，不覆盖

```javascript
// 核心逻辑
for (const [addr, info] of Object.entries(userInfo)) {
  if (!info.score) continue;  // score=0 跳过
  const prev = profiles[addr];
  profiles[addr] = {
    status: info.status || '散户',
    score: info.score,
    score_reasons: info.score_reasons || [],
    first_seen: prev ? prev.first_seen : now,
    last_seen: now,
    seen_count: prev ? prev.seen_count + 1 : 1,
    last_mint: mint
  };
}
```

## 加载逻辑（loadKnownBosses）

**文件**：`src/helius/HeliusMonitor.js`

**触发时机**：`updateHolderData()` 第一步（更新 userInfo）之后，评分计算之前（步骤1.5a）。

**加载条件**：档案库中 `score > 0` 的记录（不区分 status）

**加载效果**：在 `metricsEngine.userInfo` 上设置两个字段：
- `is_known_boss = true` → 触发 BossLogic 规则10
- `known_boss_seen_count = p.seen_count` → 规则10显示出现次数

```javascript
for (const addr of users) {
  const p = profiles[addr];
  if (p && p.score > 0) {
    userInfo[addr].is_known_boss = true;
    userInfo[addr].known_boss_seen_count = p.seen_count;
  }
}
```

## 评分流程中的档案库规则

### BossLogic 规则 10 — 历史庄家

**文件**：`src/content/BossLogic.js`

**条件**：`user.is_known_boss === true`（由 loadKnownBosses 注入）

**效果**：在当前 mint 评分基础上额外增加 `weight_known_boss`（默认 20 分）

```javascript
// 10. 历史庄家（跨 mint 档案库识别）
if (user.is_known_boss) {
    const w = config.weight_known_boss || 20;
    const times = user.known_boss_seen_count || 1;
    score += w;
    reasons.push(`历史庄家(×${times})(+${w})`);
}
```

规则10的分数和其他规则（无来源、隐藏中转等）累加后，由 `statusThreshold` 决定最终是否标为庄家。

## 完整执行流程

```
打开新 mint 页面
        ↓
updateHolderData(holders)
        ↓
1. metricsEngine.updateUsersInfo(holders)   ← 更新当前 mint 的 holder 数据
        ↓
1.5a. loadKnownBosses()                     ← 从档案库标记历史得分用户
        ↓
1.5b. detectHiddenRelays()                  ← 隐藏中转检测（有独立缓存）
        ↓
2~3. scoringEngine.calculateScores()        ← BossLogic 规则1~10 计算总分
        ↓
4. filterUsersByScore()                     ← 按 scoreThreshold 过滤显示
        ↓
5~6. updateWhaleAddresses / setFilteredUsers
        ↓
7. recalculateMetrics() → 发送数据到 UI
        ↓
7.5. saveBossProfiles()                     ← 保存 score>0 用户到档案库
```

## 状态判断阈值说明

最终 `status`（庄家/散户）由 ScoringEngine 决定：

```javascript
const isWhale = finalScore >= statusThreshold;
status: isWhale ? '庄家' : '散户'
```

`statusThreshold` 默认 **50**，可在侧边栏「状态判断分数」处调整。

**示例**：
- 用户历史得分25分（无来源10 + 隐藏中转15），保存到档案库
- 新 mint 再次出现，规则10加20 → 总分45
- 若 statusThreshold=50，状态仍为散户
- 若 statusThreshold=45，状态变为庄家

档案库中保存的 `status` 字段反映的是**保存时刻的 statusThreshold 下的判断结果**，与当前阈值设置可能不同。

## 设置界面（BossProfileManager）

**文件**：`src/content/components/BossSettingsModal.jsx`

在「庄家智能筛选配置」弹窗底部展示，包含：

| 功能 | 说明 |
|---|---|

## 注意事项

1. **BossDetector.js 策略1~4** 存在但其宿主 `MetricsEngine.detectWhales()` 未被调用，属历史代码，不参与当前评分流程。
2. **隐藏中转检测**（`detectHiddenRelays`）有独立的 `hidden_relay_{address}` 缓存，不受档案库影响，不会重复执行链上查询。
3. **saveBossProfiles** 是 fire-and-forget（不 await），不会阻塞 UI 渲染。
4. **导入时**采用合并策略，同一地址以导入数据为准（覆盖旧数据），不清空其他记录。
