helius mcp 回答：
主要接口：

getSignaturesForAddress - 标准 RPC 接口，可获取钱包的签名列表
getTransactionsForAddress - Helius 独有的增强接口（需要 Developer 计划或更高）
关于倒排序获取第一个签名：

getSignaturesForAddress 默认按时间倒序返回（最新的在前）
要获取钱包的第一个（最早的）签名，可以使用 before 参数进行分页，一直查询到最早的交易
getTransactionsForAddress 支持 sortOrder: 'asc' 参数，可以按时间正序排列（最早的在前），更容易获取第一个签名

根据搜索结果，我来详细解释 getTransactionsForAddress 接口中 filters 的高级过滤器参数：

filters - 高级过滤器详解
1. blockTime - 时间范围过滤
按区块时间戳（Unix时间戳，秒）过滤交易：

filters: {
  blockTime: {
    gte: 1735689600,   // 大于等于（Jan 1, 2025）
    gt: 1735689600,    // 大于
    lte: 1738368000,   // 小于等于（Jan 31, 2025）
    lt: 1738368000,    // 小于
    eq: 1735689600     // 等于特定时间
  }
}
用途： 获取特定时间段内的交易，如月度报告、季度分析等。

2. status - 交易状态过滤
按交易执行状态过滤：

"succeeded" - 只返回成功的交易
"failed" - 只返回失败的交易
"any" - 返回所有交易（默认）
filters: {
  status: "succeeded"  // 只获取成功交易
}
用途： 分析成功率、排除失败交易干扰、审计有效操作等。

3. tokenAccounts - 代币账户过滤
控制是否包含关联代币账户的交易：

"none" (默认) - 只返回直接涉及钱包地址的交易
"balanceChanged" (推荐) - 返回涉及钱包地址或修改钱包拥有的代币账户余额的交易
"all" - 返回涉及钱包地址或钱包拥有的任何代币账户的交易
filters: {
  tokenAccounts: "balanceChanged"  // 包含代币转账等有意义的操作
}
重要说明：

Solana上钱包不直接持有代币，而是通过关联代币账户(ATA)持有
"balanceChanged" 过滤掉垃圾交易和无关操作，只保留有意义的钱包活动
"all" 包含所有相关交易，可能包含手续费收集、委托等操作
此功能不支持2022年12月之前的历史交易
实际应用示例：

// 获取2025年1月成功的代币交易
filters: {
  blockTime: {
    gte: 1735689600,
    lte: 1738368000
  },
  status: "succeeded",
  tokenAccounts: "balanceChanged"
}

基于搜索结果，我来回答您关于同一区块内多条签名排序的问题：

同一区块内的交易排序
是的，getSignaturesForAddress 在同一区块内的多条签名也是按交易顺序排列的：

区块内排序: 同一个 slot（区块）内的交易按照它们在区块中的实际执行顺序排列
时间一致性: 虽然 blockTime 相同，但交易在区块内有明确的执行序列
分页保持: 1000条分页获取时，这个区块内顺序保持不变
具体排序逻辑
// 同一区块内的签名示例
[
  { signature: "sig1", slot: 12345, blockTime: 1640995200 }, // 区块内第1个交易
  { signature: "sig2", slot: 12345, blockTime: 1640995200 }, // 区块内第2个交易  
  { signature: "sig3", slot: 12345, blockTime: 1640995200 }, // 区块内第3个交易
  { signature: "sig4", slot: 12344, blockTime: 1640995100 }, // 上一个区块
]
排序规则总结
主要排序: 按 slot 倒序（最新区块在前）
次要排序: 同一 slot 内按交易在区块中的执行顺序
时间基准: blockTime 相同时，依然保持区块内的交易序列
实际应用建议
对于 Pump 代币监控，这个排序特性很重要：

可以准确追踪同一区块内的交易顺序
分析 MEV 或套利交易的执行序列
确保数据处理的时间一致性

存储选型：放哪里？
浏览器插件主要有两种存储方式，针对你的场景，结论非常明确：

chrome.storage.local (不推荐用于全量数据)：

限制：虽然可以通过声明 unlimitedStorage 权限解除 5MB 限制，但它是基于 JSON 序列化的。每次读取 50MB 的 JSON 都会阻塞插件的主线程，导致 UI 瞬间卡死。

IndexedDB (强烈推荐)：

理由：它是浏览器内置的非关系型数据库，支持索引（Index），适合处理万级、十万级的数据。

性能：它是异步读写，不会卡住你的插件面板或页面。

容量：通常可以占用硬盘可用空间的 20% - 50%，对于你的 Sig 记录和用户分类来说，几个 GB 都没问题。
