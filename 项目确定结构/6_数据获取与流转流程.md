# 数据获取与流转流程

## 概述

本文档描述 GMGN 插件中网络数据的拦截、获取和流转流程，以及分页获取的控制逻辑。

---

## 一、XHR 拦截（hook.js）

`hook.js` 注入到页面上下文，重写 `XMLHttpRequest` 的 `open` / `send` 方法，拦截以下 URL 的请求：

| URL 特征 | 说明 |
|---------|------|
| `/vas/api/v1/token_holders/` | 持有人数据 |
| `/vas/api/v1/token_trades/` | 交易数据 |
| `/defi/quotation/v1/trades/` | 交易数据（备用路径） |
| `/api/v1/follow/get_remark_info/` | 备注数据 |

拦截到响应后，触发唯一出口事件：

```
XHR 响应完成
  → window.dispatchEvent('HOOK_FETCH_XHR_EVENT', { url, responseBody, requestHeaders, ... })
```

> **注意**：`HOOK_FETCH_XHR_EVENT` 携带完整的 `responseBody`，是数据进入系统的唯一入口。不再单独触发 `HOOK_SIGNATURES_EVENT`（已删除，避免重复）。

---

## 二、实时 XHR 数据处理（HeliusIntegration）

`HeliusIntegration.hookFetchXhrHandler` 监听 `HOOK_FETCH_XHR_EVENT`：

```
HOOK_FETCH_XHR_EVENT (url 包含 /token_trades/)
  → 解析 responseBody，提取 trades 数组
  → 遍历每条 trade：
      signatureManager.addSignature(tx_hash, 'plugin', trade)  // 存储完整数据
  → 如果 monitor.isInitialized 且有新交易：
      processNewGmgnTrades(newTrades)  // 立即处理
```

---

## 三、分页获取流程（EXECUTE_TRADES_REFRESH）

### 触发方式

`App.jsx`（SidePanel）定时发送 `EXECUTE_TRADES_REFRESH` 消息，携带 `lastTradesUrlRef.current`（由 `HOOK_FETCH_XHR_EVENT` 首次捕获时保存）。

### 分页逻辑

```
App.jsx 发送 EXECUTE_TRADES_REFRESH (url)
  → content/index.jsx 接收
  → 加锁 isFetchingTrades = true
  → 记录 currentTradesTaskMint = getMintFromPage()
  → 循环分页：
      每次循环开始：检查 currentTradesTaskMint === getMintFromPage()（mint 变化则 break）
      fetch(currentUrl)  →  解析 trades
      window.__heliusIntegration.processFetchedTrades(trades)  // 直接调用，不走模拟事件
      nextCursor 存在则继续，每页间隔 1 秒
  → 完成后：hasFetchedFullTradesHistory = true
  → 释放锁 isFetchingTrades = false
```

### processFetchedTrades 方法（HeliusIntegration）

```
processFetchedTrades(trades)
  → 遍历 trades，检查 signatureManager 是否已有该 tx_hash
  → 新交易：addSignature + 加入 newTrades 列表
  → 如果 monitor.isInitialized 且有新交易：processNewGmgnTrades(newTrades)
  → 返回 newTradesCount（新增数量，用于重叠检测）
```

---

## 四、智能停止策略（避免重复分页）

| 场景 | 行为 |
|-----|------|
| 首次全量同步（`hasFetchedFullTradesHistory = false`） | 分页到底，不提前停止 |
| 后续定时刷新（`hasFetchedFullTradesHistory = true`） | 若 `newCount < trades.length`（有重叠），立即停止 |

`newCount` 由 `processFetchedTrades` 返回，表示本页中真正新增的交易数量。

---

## 五、Mint 切换保护

切换代币页面时，旧的分页任务会被安全终止：

```
TAB_URL_CHANGED 消息
  → currentTradesTaskMint = null   // 使旧任务失效
  → isFetchingTrades = false       // 强制释放锁
  → hasFetchedFullTradesHistory = false  // 新 mint 需重新全量同步
  → contentManager.clearData()    // 清空本地数据

HeliusIntegration（URL 变化监听）
  → checkAndInitMonitor()
  → monitor.stop()                 // 停止旧 monitor
  → this.monitor = new HeliusMonitor(newMint)  // 新 monitor，数据完全隔离
```

分页循环每次迭代开始时检查 mint，`currentTradesTaskMint = null` 后下一次循环立即 break（最多等待 1 秒的 sleep 间隔）。

## 六、完整数据流总结

```
[hook.js] XHR 拦截
  → HOOK_FETCH_XHR_EVENT（唯一出口）
  ├─ [HeliusIntegration.hookFetchXhrHandler]  实时处理（页面自然触发的 XHR）
  └─ [content/index.jsx HOOK_FETCH_XHR_EVENT 监听]
       token_holders → 发送 UI_RENDER_DATA（带 url）→ App.jsx 保存 lastGmgnUrlRef
       token_trades  → 发送 UI_RENDER_DATA（带 url）→ App.jsx 保存 lastTradesUrlRef
       get_remark_info → 触发备注全量同步

[App.jsx] 定时触发 EXECUTE_TRADES_REFRESH（使用 lastTradesUrlRef）
  → [content/index.jsx] 分页获取
  → window.__heliusIntegration.processFetchedTrades(trades)
  → [HeliusIntegration] signatureManager + processNewGmgnTrades
  → [HeliusMonitor] MetricsEngine + ScoringEngine
  → HELIUS_DATA_UPDATE 事件
  → [content/index.jsx] 发送 UI_RENDER_DATA
  → [App.jsx] UI 更新
```
