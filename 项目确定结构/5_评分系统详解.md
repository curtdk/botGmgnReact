# 庄家识别评分系统详解

## 一、整体流程

```
GMGN Holder API 快照到达
        │
        ▼
HeliusIntegration.updateGmgnHolders()
        │
        ▼
HeliusMonitor.updateHolderData(holders)
        ├─ Step 1: MetricsEngine.updateUsersInfo()
        │          把 Holder 字段（funding_account / ui_amount 等）
        │          merge 进 traderStats[address]
        │
        ├─ Step 2: detectHiddenRelays()（仅当 enable_hidden_relay=true）
        │          只对 有 funding_account 且未检测过 的用户做慢速翻页
        │          → 设置 has_hidden_relay / hidden_relay_conditions
        │
        ├─ Step 3: ScoringEngine.calculateScores()
        │          对所有 traderStats 中的用户运行 9 条规则累加分数
        │          → scoreMap { address: { score, reasons, isWhale, status } }
        │          → whaleAddresses Set（score >= statusThreshold 的地址）
        │
        ├─ Step 4: 写回 traderStats（score / status / score_reasons）
        │
        ├─ Step 5: _persistUserScores()（异步写入 IndexedDB users 表）
        │
        ├─ Step 6: filterUsersByScore() → filteredUsers
        │          score < scoreThreshold 的地址加入集合
        │
        └─ Step 7: 更新 MetricsEngine 的两个过滤集合
                   metricsEngine.updateWhaleAddresses(whaleAddresses)
                   metricsEngine.setFilteredUsers(filteredUsers)
                   → 触发 4 大参数重新计算
```

---

## 二、两个核心阈值

| 阈值 | 默认值 | 作用 | 调整方式 |
|---|---|---|---|
| `statusThreshold` | 50 | `score >= threshold` → 加入 `whaleAddresses`（庄家） | 插件 Score< 设置 |
| `scoreThreshold` | 100 | `score < threshold` → 加入 `filteredUsers`（参与4大参数计算）| 插件 Score< 设置 |

**whaleAddresses（庄家集合）**：在 `processTransaction()` 入口处过滤，庄家的交易不进入实时交易列表，也不参与任何指标计算。

**filteredUsers（白名单集合）**：在 `getMetrics()` 计算 4 大参数时过滤，只统计集合内用户的持仓数据。

---

## 三、9 条评分规则

规则由 `BossLogic.calculateUserScore()` 按顺序执行，各规则独立累加，互不影响。

---

### 规则 1 — 无资金来源

| 项目 | 内容 |
|---|---|
| 配置开关 | `enable_no_source` |
| 默认权重 | `weight_no_source`（默认 10） |
| 触发条件 | `!user.funding_account`（Holder 快照中 Funding Address 为空） |
| reason 标记 | `无来源(+N)` |

**说明**：funding_account 来自 GMGN Holder API 的 `native_transfer.from_address` 字段，为空表示该钱包无可追溯的链上资金来源。

---

### 规则 2 — 同源账户

| 项目 | 内容 |
|---|---|
| 配置开关 | `enable_same_source` |
| 默认权重 | `weight_same_source`（默认 10） |
| 触发条件 | 该用户的 `funding_account` 被 ≥ `same_source_n`（默认5）个不同用户共享，且该 funding_account 不在排除名单 `same_source_exclude` 中 |
| reason 标记 | `同源(N人)(+N)` |

**说明**：同一个资金来源地址（如同一个转账钱包）下辖多个持仓地址，是批量建号的特征。排除名单用于过滤交易所充值地址等合规来源。

---

### 规则 3 — 时间聚类

| 项目 | 内容 |
|---|---|
| 配置开关 | `enable_time_cluster` |
| 默认权重 | `weight_time_cluster`（默认 10） |
| 触发条件 | 在 `time_cluster_j`（默认1秒）时间窗口内，有 ≥ `time_cluster_n`（默认5）个用户的首次买入时间紧密相邻 |
| 数据来源 | `user.created_at` 或 `user.open_timestamp` |
| reason 标记 | `时间聚集(+N)` |

**说明**：多个地址在极短时间内同步买入，是脚本批量操作的典型特征。

---

### 规则 4 — Gas 费用异常（低 Gas）

| 项目 | 内容 |
|---|---|
| 配置开关 | `rule_gas.enabled` |
| 默认权重 | `rule_gas.weight`（默认 10） |
| 触发条件 | `user.max_gas_fee > 0` 且 `user.max_gas_fee < rule_gas.threshold`（默认 0.01） |
| reason 标记 | `LowGas(N)(+N)` |

**说明**：庄家常用私有节点或脚本，Gas 费用极低，与正常用户的随机波动不同。

---

### 规则 5 — 买入金额相似

| 项目 | 内容 |
|---|---|
| 配置开关 | `rule_amount_sim.enabled` |
| 默认权重 | `rule_amount_sim.weight`（默认 10） |
| 触发条件 | 在以 `rule_amount_sim.range`（默认100 USD）为步长的桶内（本桶±1桶），累计用户数 ≥ `rule_amount_sim.count`（默认5） |
| 数据来源 | `user.total_buy_u` |
| reason 标记 | `金额相似(N人)(+N)` |

**说明**：多个地址买入金额高度相近（如均为 500 USD），是脚本统一配置的特征。

---

### 规则 6 — 大额持仓

| 项目 | 内容 |
|---|---|
| 配置开关 | `rule_large_holding.enabled` |
| 默认权重 | `rule_large_holding.weight`（默认 10） |
| 触发条件（AND 模式） | `rank ≤ 总持有人数 × top_pct%` 且 `usd_value > min_usd` |
| 触发条件（OR 模式） | 满足排名或金额任一 |
| reason 标记 | `大额持仓(+N)` |

---

### 规则 7 — SOL 余额相似

| 项目 | 内容 |
|---|---|
| 配置开关 | `rule_sol_balance.enabled` |
| 默认权重 | `rule_sol_balance.weight`（默认 10） |
| 触发条件 | 在以 `rule_sol_balance.range`（默认0.1 SOL）为步长的桶内（本桶±1桶），累计用户数 ≥ `rule_sol_balance.count`（默认3） |
| 数据来源 | `user.sol_balance` |
| reason 标记 | `余额相似(N人)(+N)` |

**说明**：多个地址当前 SOL 余额高度一致，是统一充值后统一消耗的特征。

---

### 规则 8 — 资金来源时间聚类

| 项目 | 内容 |
|---|---|
| 配置开关 | `rule_source_time.enabled` |
| 默认权重 | `rule_source_time.weight`（默认 10） |
| 触发条件 | 在 `rule_source_time.diff_sec`（默认10秒）窗口内，有 ≥ `rule_source_time.count`（默认2）个用户的资金转入时间紧密相邻 |
| 数据来源 | `user.native_transfer.timestamp` |
| reason 标记 | `同源时间(N): addr1,addr2...(+N)` |

**说明**：不同资金来源地址但转入时间极为接近，说明背后可能由同一主账户统一调度。

---

### 规则 9 — 无资金来源/隐藏中转（双条件短路）

| 项目 | 内容 |
|---|---|
| 配置开关 | `enable_hidden_relay` |
| 默认权重 | `weight_hidden_relay`（默认 15） |

**两个条件，满足其一即得分，且条件1优先（短路）**：

**条件1（快，零延迟）**：`!user.funding_account`
- Funding Address 为空 → 直接成立，**不做任何慢速检测**
- reason 标记：`无资金来源/中转(+N)`

**条件2（慢，仅当 funding_account 存在时触发）**：`user.has_hidden_relay === true`
- 由 `detectHiddenRelays()` 异步检测：翻页拉取用户全部历史 sig，找到最早一笔 tx，判断是否含 `Create + CloseAccount` 指令（资金通过中间账户中转到此钱包的典型模式）
- 检测结果写入 IndexedDB（key = address，跨 mint 复用），下次同一地址不再重复检测
- reason 标记：`隐藏中转[指令类型](+N)`

**detectHiddenRelays() 跳过条件（按顺序检查）**：
```
1. !userInfo[u].funding_account → 条件1已覆盖，直接跳过
2. ud.hiddenRelayCheckedAt 存在 → 已检测过，读缓存恢复状态
3. ud.score !== undefined     → 已有完整评分，跳过
```

---

### 手动标记（附加分）

| 项目 | 内容 |
|---|---|
| 来源 | `manualScores[address] === '庄家'` |
| 附加分 | 固定 +10 |
| reason 标记 | `手动标记(+10)` |
| 持久化 | IndexedDB users 表 + `manualScores` 内存缓存 |

手动标记不是独立规则，而是在所有规则计算完成后追加到 `finalScore` 上。

---

## 四、最终状态判断

```
规则1~9 累加 + 手动标记
    = finalScore

finalScore >= statusThreshold（默认50）
    → isWhale = true
    → status = '庄家'
    → 加入 whaleAddresses

finalScore < statusThreshold
    → isWhale = false
    → status = '散户'

finalScore < scoreThreshold（默认100）
    → 加入 filteredUsers（参与4大参数计算）
```

---

## 五、评分触发时机

| 触发来源 | 方法 | 写入 IndexedDB | 备注 |
|---|---|---|---|
| GMGN Holder API 快照到达 | `updateHolderData()` | 写入（score/status/reasons） | 主评分路径，含隐藏中转检测 |
| WS 实时新交易 | `_scheduleQuickScore()`（500ms 防抖） | 不写入 | 快速刷新内存状态，不做慢速检测 |
| 阈值变更 `setScoreThreshold()` | 直接从内存分数重算 `filteredUsers` | 不写入 | 无需重新评分 |
| 阈值变更 `setStatusThreshold()` | 仅更新内存值 | 不写入 | 下次评分时生效 |

## 六、数据流向图

```
GMGN Holder API
    │ funding_account / ui_amount / sol_balance / native_transfer 等
    ▼
traderStats[address]（内存）
    │
    ├── BossLogic 9条规则 → finalScore
    │       │
    │       └── manualScores[address] → +10（可选）
    │
    ├── finalScore >= statusThreshold → whaleAddresses（庄家，拦截交易入口）
    ├── finalScore < scoreThreshold  → filteredUsers（参与4大参数计算）
    │
    └── IndexedDB users 表（异步持久化）
            ├── score
            ├── status
            ├── reasons
            └── hiddenRelayCheckedAt / hiddenRelay / hiddenRelayConditions
```

修改后（与 deleteUsersAbove 对称）：


u.score !== undefined && u.score < scoreThreshold && !u.manualScore
→ 只删除有明确分数且 score < 阈值、非手动标记的用户

现在两个删除函数行为一致：

清除 score < X → 只删除已评分且分数低于 X 的用户
清除 score ≥ X → 只删除已评分且分数高于等于 X 的用户
手动标记的用户（manualScore）两种情况都不会被删除
