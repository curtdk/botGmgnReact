# 数据源架构

## 概述

本项目使用 **HeliusMonitor** 作为唯一的数据处理和存储中心。所有数据从 GMGN API 获取后，统一由 HeliusMonitor 处理、评分、过滤，然后分发给 UI。

## 数据流架构

```
GMGN API (holders/trades)
    ↓
[index.jsx] 拦截 API 响应
    ↓ normalize (统一字段名)
    ↓
[HeliusIntegration] 协调器
    ↓ 加载配置 (boss_config, score_threshold, status_threshold)
    ↓
[HeliusMonitor] 数据处理中心
    ├─ [MetricsEngine] 存储用户信息和交易统计
    ├─ [ScoringEngine] 计算用户分数
    └─ [过滤逻辑] 根据 Score< 阈值过滤
    ↓
[HeliusIntegration.sendDataToSidepanel()]
    ↓
UI (Sidepanel) 显示用户列表
```

## 核心组件

### 1. HeliusMonitor (数据处理中心)

**文件**: `/src/helius/HeliusMonitor.js`

**职责**:
- 唯一的数据存储和处理中心
- 接收 GMGN holder 和 trade 数据
- 调用 ScoringEngine 计算分数
- 根据 Score< 阈值过滤用户
- 计算指标（已落袋、本轮下注等）

### 2. MetricsEngine (数据存储引擎)

**文件**: `/src/helius/MetricsEngine.js`

**职责**:
- 存储所有用户的完整信息 (userInfo)
- 存储交易统计数据 (traderStats)
- 管理过滤用户列表 (filteredUsers)
- 管理庄家地址列表 (whaleAddresses)

**关键方法**:
- `updateUserInfo(holderData)` - 更新单个用户信息
- `updateUsersInfo(holdersArray)` - 批量更新用户信息
- `setFilteredUsers(filteredUsers)` - 设置过滤后的用户列表
- `updateWhaleAddresses(whaleAddresses)` - 更新庄家地址列表

### 3. ScoringEngine (评分引擎)

**文件**: `/src/helius/ScoringEngine.js`

**职责**:
- 使用 BossLogic 的 8 个评分规则计算用户分数
- 处理手动标记 (+10 分)
- 返回 scoreMap 和 whaleAddresses

**评分规则** (来自 BossLogic):
1. 无资金来源 (+10)
2. 同源地址 (+10)
3. 时间聚集 (+10)
4. 高持仓占比 (+10)
5. 高买入金额 (+10)
6. 早期买入 (+10)
7. 多次买入 (+10)
8. 手动标记 (+10)

**返回数据结构**:
```javascript
{
  scoreMap: Map([
    ["地址1", { score: 20, reasons: ["无来源(+10)", "同源(+10)"] }],
    ["地址2", { score: 0, reasons: [] }]
  ]),
  whaleAddresses: Set(["地址1"])  // score >= statusThreshold 的用户
}
```

### 4. HeliusIntegration (协调器)

**文件**: `/src/content/HeliusIntegration.js`

**职责**:
- 接收 GMGN API 数据
- 加载配置 (boss_config, score_threshold, status_threshold)
- 传递配置给 HeliusMonitor
- 监听配置变化并实时更新
- 发送过滤后的数据给 UI

**关键方法**:
- `updateGmgnHolders(holders)` - 接收 GMGN holder 数据
- `updateGmgnTrades(trades)` - 接收 GMGN trade 数据
- `sendDataToSidepanel()` - 发送数据给 UI
- `init()` - 初始化并加载配置

## 数据处理流程

### Holder 数据处理流程

1. **GMGN API 响应** → index.jsx 拦截
2. **Normalize** → 统一字段名 (owner, ui_amount, etc.)
3. **HeliusIntegration.updateGmgnHolders()** → 接收数据
4. **HeliusMonitor.updateHolderData()** → 开始处理
   - 步骤 1: MetricsEngine.updateUsersInfo() → 更新 userInfo
   - 步骤 2: ScoringEngine.calculateScores() → 计算分数
   - 步骤 3: 存储分数到 userInfo
   - 步骤 4: 过滤用户 (score < threshold)
   - 步骤 5: 更新庄家地址
   - 步骤 6: 设置过滤后的用户列表
   - 步骤 7: 重新计算指标
5. **HeliusIntegration.sendDataToSidepanel()** → 发送给 UI
6. **UI 显示** → 用户列表

### 配置变化处理流程

1. **用户修改 Score< 阈值** → Chrome Storage 更新
2. **HeliusIntegration 监听变化** → storage.onChanged
3. **更新 HeliusMonitor 配置** → monitor.setScoreThreshold()
4. **重新过滤用户** → filterUsersByScore()
5. **发送更新后的数据** → sendDataToSidepanel()
6. **UI 实时更新** → 用户列表刷新

## 关键设计原则

1. **唯一数据源**: HeliusMonitor 是唯一的数据处理中心，避免数据冲突
2. **提前过滤**: 在数据源中根据 Score< 阈值过滤，然后发送给 UI
3. **配置持久化**: 所有配置从 Chrome Storage 加载，确保一致性
4. **实时同步**: 配置变化时立即更新并重新过滤数据

## 注意事项

1. **地址字段统一**: GMGN API 可能返回 `owner`、`address` 或 `wallet_address`，需要在 index.jsx 中统一为 `owner`
2. **默认值一致**: score_threshold 默认值为 100，status_threshold 默认值为 50
3. **过滤逻辑**: 只有 score < threshold 的用户才会被显示和计算指标
4. **状态判断**: score >= statusThreshold 的用户状态为"庄家"，否则为"散户"
